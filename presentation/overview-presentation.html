The Problem

  • Real-world code is often incomplete — legacy migrations, decompiled binaries, partial extracts, missing dependencies
  • Traditional tools halt at the first gap — one missing import, one unknown API call, and analysis stops entirely
  • Asking an LLM to "just figure it out" is 100–1,000x slower than running code directly — and most code paths don't need an LLM at all
What if we could analyse and run code deterministically wherever possible, and only call an LLM at the exact points where information is genuinely missing?

The Solution

RedDragon is a multi-language code analysis engine that:

Parses source code
16 languages, one unified representation
Analyses data flow
Which variables depend on which?
Executes deterministically
Runs concrete code locally, fast
Fills gaps with LLMs
Only for genuinely unknown values

The key: the LLM is a last resort, not the default. Most code runs without it.

Smart LLM Usage

RedDragon uses LLMs in two targeted ways:

1. As a Translator

For languages without a built-in parser, the LLM translates source code into RedDragon's internal format.

This means any programming language can be supported — even exotic or proprietary ones.

2. As a Gap-Filler

When executing code that calls an unknown API or missing function, the LLM provides a plausible result so execution can continue.

Everything else runs locally at native speed — no LLM needed.

Supports 4 LLM providers: Claude, OpenAI, Ollama (local), and HuggingFace — all through a single unified interface.

  • A simple function like factorial(5) takes ~67 microseconds locally vs. ~33 seconds if every step went through an LLM
  • Most real code is fully concrete at any given call site — only genuinely missing information needs an LLM
  • LLMs also cost money per call — minimising calls reduces both latency and cost

Graceful Degradation

RedDragon never crashes on unknown code. It degrades gracefully:

Complete code → fully deterministic execution, zero LLM calls
Partially complete code → runs what it can, marks unknowns as symbolic placeholders
Code with missing dependencies → asks an LLM only at the exact point of the gap
Unsupported language → LLM translates the entire file, then analysis proceeds normally

One unknown doesn't block the rest. Analysis and execution continue around the gaps.
